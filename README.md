# Chongqing Monopoly Game / 大富翁游戏——重庆地图版

一个基于 **C++17、Visual Studio 2022、Win32 API 与 GDI 绘图** 编写的图形化大富翁小游戏。项目以传统大富翁“掷骰前进、购买地产、建造房屋、收取过路费、机会事件、银行收益、监狱暂停、机场跳转、资金不足抵押、破产淘汰、限时结算”为主要规则，同时结合重庆区县地名设计地图，适合作为 C++ 课程设计、Win32 图形界面练习和小型游戏状态机项目参考。

## 2. 项目简介

本项目实现了一个本地运行的图形化大富翁游戏。玩家进入游戏后，可以选择玩家数量、初始资金和游戏限时。系统会根据玩家数量自动划分队伍，并通过开局掷骰决定队伍行动顺序。正式游戏开始后，当前玩家点击骰子进行移动，根据落点触发购买地产、建造房屋、缴纳过路费、抽取机会卡、进入监狱、使用机场或领取银行利息等事件。

游戏采用 Win32 原生窗口实现，界面由 GDI 绘制。棋盘为 34 格环形地图，地块名称使用重庆相关区县名称。项目代码将游戏规则、界面绘制、数据模型、常量配置和日志输出分成多个模块，整体结构比把所有逻辑写在一个 `main.cpp` 中更清晰，便于课程检查、答辩说明和后续维护。

## 3. 已实现功能

### 3.1 开局设置

- 支持选择玩家数量：2 人、3 人、4 人、6 人、8 人、9 人。
- 支持选择每人初始资金：3000 元、5000 元、8000 元、10000 元。
- 支持选择游戏限时：5 分钟、10 分钟、15 分钟、20 分钟、30 分钟。
- 根据玩家数量自动组队：
  - 2、3、4 人：每人一队。
  - 6、8 人：每 2 人一队。
  - 9 人：每 3 人一队。
- 设置确认后进入开局掷骰阶段，避免游戏中途重新修改人数或资金。

### 3.2 行动顺序与回合控制

- 每支队伍开局掷一次骰子决定行动顺序。
- 骰子点数高的队伍先行动。
- 点数相同时，程序自动进行隐藏加掷作为排序依据，避免平局导致流程卡住。
- 每支队伍内部成员轮流行动。
- 当前行动玩家会在地图上闪烁显示。
- 右侧信息区显示当前队伍、当前玩家、剩余时间、骰子点数、移动结果和最近操作信息。

### 3.3 地图与地块

游戏地图共 34 格，按顺时针方向移动，包含以下类型：

- **起点**：经过或到达起点时，当前玩家获得起点奖励。
- **房产**：可购买、可建房、可收取过路费。
- **机会**：随机抽取机会卡，并执行对应效果。
- **监狱**：到达后，当前队伍下一次行动暂停。
- **机场**：到达后，可以点击地图任意地块作为飞行目的地。
- **银行**：到达后，当前玩家获得手中现金十分之一的利息。

### 3.4 房产与建房系统

- 无主房产可以购买。
- 已购买房产会记录所属队伍。
- 到达本队房产时，可以选择建造房屋。
- 每块房产最多建造 3 栋房屋。
- 建房费用随已有房屋数量递增。
- 到达其他队伍房产时，需要支付过路费。
- 过路费随房屋数量增加。
- 房价、建房费和过路费会根据初始资金进行比例缩放。
- 队伍收到的过路费会在队员之间平分。

### 3.5 机会卡系统

当前版本内置 10 张机会卡，包括：

- 项目分红：获得现金。
- 车辆维修：支付维修费。
- 回到起点：移动到起点并获得起点奖励。
- 违规停车：前往监狱，并让全队暂停一次。
- 理财收益：按当前现金比例获得收益。
- 房屋维护：按本队房屋数量支付维护费。
- 银行贵宾：前往银行并领取利息。
- 商圈活动：获得现金。
- 临时罚款：支付罚款。
- 小额返利：按当前现金比例获得返利。

机会卡数据集中保存在 `constants.h` 中，后续增加新卡牌时不需要大幅修改主流程。

### 3.6 抵押与破产系统

当玩家现金不足时，系统会根据付款类型进入不同处理流程：

- 购买房产、建房属于非强制付款，资金不足时可以放弃本次操作。
- 过路费、机会卡罚款、房屋维护费属于强制付款。
- 强制付款现金不足时，可以选择抵押本队房产。
- 抵押金额为该地块总投入的 30%。
- 抵押后，房产和房屋归还银行。
- 抵押所得由本队成员平分。
- 如果玩家没有手动选择抵押房产，程序可以自动抵押本队抵押价值最低的房产。
- 若仍无法付清强制付款，并且没有可继续抵押的房产，则当前队伍破产淘汰。
- 队伍淘汰后，其玩家现金清零、棋子移出地图、房产归还银行，并从行动顺序中删除。

### 3.7 游戏结束与结算

当前版本已经实现游戏结束阶段 `game_over`，主要有两种结束方式：

1. **淘汰结束**：当只剩一支队伍未被淘汰时，游戏直接结束。
2. **限时结束**：游戏限时到达后，按照财富折算排名。

最终结算公式为：

```text
队伍最终财富 = 队伍玩家手上剩余现金之和 × 1 + 所买地产总价值 × 1.2
```

其中，地产总价值按“地价 + 已建房投入”计算。被淘汰队伍会在结算中显示为 `out`。

### 3.8 图形界面与交互

- 使用 Win32 API 创建窗口。
- 使用 GDI 绘制地图、地块、棋子、按钮、骰子和中央信息区。
- 窗口标题为“重庆大富翁游戏”。
- 窗口大小为 1360 × 900。
- 字体使用 Microsoft YaHei UI。
- 支持鼠标点击按钮完成设置、购买、建房、抵押和取消等操作。
- 支持点击骰子区域投掷骰子。
- 骰子投掷时带有简单动画。
- 支持机场阶段点击地图地块选择目的地。
- 非法点击会在信息区和日志中记录原因。

### 3.9 日志系统

- 每局游戏启动时自动创建日志文件。
- 日志目录为 `result/`。
- 日志文件名格式为：`YYYYMMDD_HHMMSS.txt`。
- 日志使用 GBK 无 BOM 编码，中文 Windows 记事本可直接打开。
- 日志记录内容包括：游戏开始、开局设置、开局掷骰、行动顺序、玩家移动、购买房产、建造房屋、支付过路费、机会卡、抵押、破产淘汰、限时结算等。

## 4. 运行环境

运行本项目需要以下环境：

1. Windows 操作系统。
2. Visual Studio 2022。
3. 已安装“使用 C++ 的桌面开发”组件。
4. Windows SDK。
5. 支持 C++17 的 MSVC 编译器。

工程文件中使用的是 Win32 项目配置，字符集为 Unicode，编译参数包含：

```text
/source-charset:.936 /execution-charset:.936
```

这表示源码和执行字符集按 GBK 处理。项目不依赖 EasyX、Qt、SDL、Unity 等第三方图形库。

注意：工程文件中的平台工具集写为 `v145`。如果本地 Visual Studio 没有对应工具集，可以在项目属性中改为本机已安装的 MSVC 工具集后再编译。

## 7. 模块与类关系

### 7.1 `main.cpp`

`main.cpp` 是程序入口，只负责创建 `game_ui` 对象并调用 `run()`，把控制权交给 Win32 窗口消息循环。入口文件不直接处理游戏规则，保持了结构简洁。

### 7.2 `game_ui`

`game_ui` 是界面层，主要负责：

- 注册并创建 Win32 主窗口。
- 处理 `WM_PAINT`、`WM_LBUTTONDOWN`、`WM_TIMER`、`WM_DESTROY` 等消息。
- 绘制地图、地块、玩家棋子、按钮、骰子和信息区。
- 判断鼠标点击命中的按钮、骰子或地图地块。
- 启动骰子动画。
- 通过定时器刷新当前玩家闪烁效果。
- 每秒检查游戏剩余时间。
- 将用户点击转换成动作，交给 `game_engine` 执行。

### 7.3 `game_engine`

`game_engine` 是游戏规则核心，主要负责：

- 管理当前游戏阶段。
- 保存玩家、队伍、房产、行动顺序、当前回合和待付款信息。
- 处理开局设置。
- 创建玩家和队伍。
- 执行开局掷骰排序。
- 执行正式回合移动。
- 判断落点地块类型。
- 处理购买房产、建房、过路费、机会卡、监狱、机场、银行等规则。
- 处理抵押、自动抵押、破产淘汰。
- 处理限时结束和最终排名。
- 生成界面需要显示的文字和按钮选项。

### 7.4 `game_logger`

`game_logger` 负责日志输出，主要负责：

- 创建 `result` 文件夹。
- 按当前时间生成日志文件名。
- 将宽字符中文转换为 GBK 编码。
- 写入带时间戳的游戏日志。

### 7.5 `models`

`models.h` 定义游戏运行过程中使用的数据结构，例如：

- `game_phase`：游戏阶段。
- `action_id`：按钮动作类型。
- `action_option`：按钮显示内容和附带参数。
- `player`：玩家状态。
- `team`：队伍状态。
- `property_state`：房产归属和房屋数量。

### 7.6 `constants`

`constants.h` 定义静态游戏配置，例如：

- 地图总格数。
- 起点、监狱、机场、银行位置。
- 最大建房数量。
- 可选玩家数量。
- 可选初始资金。
- 可选游戏时间。
- 地块名称、类型、房价、建房费、基础过路费。
- 机会卡标题、说明和效果。

## 8. 主要算法与实现思路

### 8.1 游戏状态机

项目使用 `game_phase` 枚举管理游戏流程：

```cpp
enum class game_phase {
    setup,
    opening_roll,
    turn_ready,
    awaiting_property_decision,
    awaiting_build_decision,
    awaiting_mortgage_consent,
    awaiting_mortgage_selection,
    awaiting_mortgage_confirm,
    awaiting_airport_destination,
    game_over
};
```

不同阶段只允许执行对应操作。例如设置阶段只能选择人数、资金和时间；购买阶段只能选择购买或放弃；抵押阶段只能选择抵押房产；机场阶段只能点击地图目的地或取消机场移动。这样可以避免非法点击破坏游戏流程。

### 8.2 自动组队规则

项目根据玩家总数自动计算每队人数：

- 6 人、8 人：每队 2 人。
- 9 人：每队 3 人。
- 其他人数：每人一队。

玩家编号从 0 开始存储，界面显示时转换为 1 号玩家、2 号玩家等。

### 8.3 开局掷骰排序

每支队伍开局掷一次骰子，系统按点数从大到小生成行动顺序。如果点数相同，程序会生成隐藏加掷点数作为第二排序依据；若隐藏加掷仍相同，则按队伍编号排序。这样既保留随机性，又避免手动重复掷骰导致流程复杂。

### 8.4 环形地图移动

地图共有 34 格。玩家移动时通过取模运算实现环形循环：

```cpp
next = (current_position + dice_value) % board_cell_count;
```

如果移动过程中跨过起点，则发放起点奖励。起点奖励为每人初始资金的十分之一。

### 8.5 金额缩放

项目以 5000 元开局为基础价格。若玩家选择 3000、8000 或 10000 元开局，房价、建房费和过路费会按比例缩放，并取整到 10 元单位。这样可以让不同初始资金下的游戏节奏保持相对平衡。

### 8.6 房产收益计算

每块房产用 `property_state` 保存所属队伍和房屋数量：

```cpp
struct property_state {
    int owner_team = -1;
    int houses = 0;
};
```

`owner_team = -1` 表示无人购买。过路费按照基础过路费和房屋数量计算：

```text
当前过路费 = 基础过路费 × (房屋数量 + 1)
```

### 8.7 机会卡随机事件

机会卡存放在 `chance_cards` 常量数组中。到达机会格后，程序随机抽取一张机会卡，并根据 `chance_effect` 执行对应逻辑。该设计将卡牌数据和主流程分离，后续扩展新机会卡比较方便。

### 8.8 抵押与破产处理

当玩家现金不足时，系统会保存一份 `pending_payment`，用于记录待支付金额、付款类型、关联地块和收款队伍。之后根据付款类型判断是否必须付款：

- 非强制付款：可失败后结束回合。
- 强制付款：需要通过抵押尽量补足金额。

如果无力继续支付，程序会先处理当前玩家剩余现金，再淘汰当前队伍。淘汰后会清理该队玩家、房产和行动顺序，保证后续回合继续正常运行。

### 8.9 限时结算

游戏启动后开始计时，界面每秒刷新剩余时间。时间结束后进入 `game_over` 阶段，并按现金和地产价值计算最终排名。地产按 1.2 倍折算，使购买与建房在最终结算中有更高权重。

## 9. 变量命名与代码风格

本项目整体采用英文语义命名，并保留大量中文注释，方便课程检查和后续维护。

- 类名使用小写加下划线：`game_engine`、`game_ui`、`game_logger`。
- 枚举名使用小写加下划线：`game_phase`、`cell_kind`、`chance_effect`、`action_id`。
- 成员变量以 `_` 结尾：`phase_`、`players_`、`teams_`、`turn_order_`、`last_dice_`。
- 常量使用语义化命名：`board_cell_count`、`start_index`、`jail_index`、`airport_index`、`bank_index`。
- 函数命名以动词或动宾结构为主：`roll_dice()`、`process_arrival()`、`start_payment()`、`mortgage_property()`、`finish_game_by_time()`。
- UI 代码和游戏规则代码分离，避免所有逻辑集中在入口文件。

## 11. 未完成内容与后续可优化方向

当前项目已经可以完成单局运行和最终结算，但仍有以下可优化方向：

1. **增加重新开始功能**
   当前游戏结束后主要展示结算结果，后续可以加入“重新开始”“返回设置界面”等按钮。
2. **增加存档与读档功能**
   当前只有日志记录，不能保存当前局面后下次继续。后续可以把玩家、队伍、房产、行动顺序和剩余时间序列化到文件。
3. **完善破产资产清算规则**
   当前破产后房产直接归还银行。后续可以设计为转移给债权队伍、拍卖或按比例赔付。
4. **增加赎回抵押机制**
   当前抵押后房产和房屋直接归还银行。后续可以改成“抵押状态”，允许支付本金和利息赎回。
5. **增加电脑玩家 AI**
   当前操作依赖人工点击。后续可以增加自动购买、自动建房、自动抵押和策略判断。
6. **增加更多机会卡**
   当前已有 10 张机会卡，后续可以增加交换位置、随机传送、强制拍卖、集体奖励、地产升值等效果。
7. **优化界面美术与动画**
   当前主要使用 GDI 基础图形绘制。后续可以增加图片资源、地图背景、按钮动效、骰子动画和更明显的地块图标。
8. **增加规则配置文件**
   目前地图和机会卡写在 `constants.h` 中。后续可以改成 JSON、CSV 或文本配置，方便不改代码就能调整地图。
9. **补充单元测试或规则测试用例**
   对金额缩放、过路费、抵押、破产、限时结算等逻辑增加测试，可以降低后续修改时出错的概率。
